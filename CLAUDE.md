# Beach Volleyball Tournaments — Czech Republic

## What this is
A landing page showcasing beach volleyball tournaments in the Czech Republic.

## Who I am
I'm a product manager learning to code. I understand product/UX thinking well,
but I'm new to the mechanics of building software. Treat me like a smart
beginner: don't assume I know jargon, but don't over-explain things I've
already seen explained once.

## Tech stack
Plain HTML, CSS, and JavaScript. No frameworks, no build tools, no npm.
Reasons: fewer moving parts to learn at once, and it's easy to see exactly
how the browser turns code into a page.

- `index.html` — structure/content of the page
- `styles.css` — visual styling
- `script.js` — interactivity (only added once we actually need it)

Keep this structure simple. Don't introduce a bundler, framework, or package
manager unless I explicitly ask for one.

## How we work together
- **One feature at a time.** Build and finish one small piece before moving
  to the next. Don't bundle multiple features into one change.
- **Explain what you build, in simple language.** After making a change,
  briefly explain: what you added, why it works the way it does, and any new
  concept involved (e.g. what a CSS flexbox is, what an HTML `<section>` is
  for). Keep explanations short and plain — a sentence or two per concept,
  not a lecture.
- **Show, don't just tell.** When a change is visual, mention how to view it
  (e.g. open `index.html` in a browser) so I can see the result myself.
- **Prefer boring, readable code** over clever tricks. This is a learning
  project — clarity beats cleverness.
- **Ask before adding new tools or dependencies.** Since the whole point is
  learning the fundamentals, don't quietly pull in a library to solve
  something we could do with plain HTML/CSS/JS.
